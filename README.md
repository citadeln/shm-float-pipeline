# Высокопроизводительный канал сжатых данных

Данный проект реализует высокопроизводительный канал передачи сжатых данных через **разделяемую память POSIX** между процессами‑производителем (`Producer`) и процессом‑потребителем (`Consumer`). Исходные `float`‑данные считываются из файла, квантуются в `int16_t`, передаются по общему кольцевому буферу и затем восстанавливаются на стороне потребителя с измерением потерь.

## Описание приложения

- **Производитель (`producer`)**  
  Читает бинарный файл с `float`‑данными, квантует их в `int16_t` заданным масштабом, группирует в пакеты с заголовком `ChunkHeader` и передаёт через разделяемую память.
- **Потребитель (`consumer`)**  
  Получает пакеты из разделяемой памяти, декодирует `int16_t` обратно в `float` и записывает восстановленные данные в выходной бинарный файл.
- **Юнит‑тесты**  
  Проверяют корректность работы квантователя и кольцевого буфера.

## Требования

### Системные требования

- Linux или любая POSIX‑совместимая система с поддержкой `shm_open`, `sem_open`, `mmap`.
- Доступ к `/dev/shm` (стандартный `tmpfs`‑раздел для разделяемой памяти).

### Зависимости

- Компилятор C++20 (`g++` или `clang++`).
- `libpthread` (POSIX threads).
- `librt` (для `mmap`/`mlock` и семафоров, где требуется).
- CMake ≥ 3.16.
- Google Test (для `tests/test_utils`).

## Сборка проекта
```bash
./scripts/build.sh
```
Скрипт автоматически:
- создаёт каталог `build`,
- вызывает `cmake` с необходимыми флагами (C++20, флаги компилятора),
- запускает `make` с параллельной сборкой (`-j`).

## Запуск проекта
```bash
./scripts/run.sh
```
Скрипт:
- собирает проект (если ещё не собран);
- запускает `producer` и `consumer` параллельно;
- передаёт им входной файл `../data/sample.bin` и записывает результат в `../data/output.bin`;
- после завершения процессов запускает юнит‑тесты из `test_utils`.

Типичный вывод:

```text
Producer: 50ms, ratio: 0.500
Consumer: 45ms, loss: 0.00%
[==========] Running 3 tests from 2 test suites.
...
```

## Метод синхронизации процессов
Для синхронизации процессов‑производителя и‑получателя используется **разделяемая память POSIX (`shm_open`)** и **именованные семафоры (`sem_open`)**.

### Используемые объекты

- `/test_channel` — общий сегмент разделяемой памяти, в котором расположены:
  - заголовок `Header` (голова, хвост, ёмкость, `magic`),
  - кольцевой буфер данных (`data_`).
- `/test_empty` — семафор, отслеживающий **количество свободных ячеек** в буфере.
- `/test_full` — семафор, отслеживающий **количество заполненных ячеек** в буфере.

### Порядок работы

1. **Инициализация**  
   Производитель вызывает `InitProducer`, создаёт:
   - `shm_open("/test_channel", ...)`,
   - `sem_open("/test_empty", ...)`, `sem_open("/test_full", ...)`.
2. **Ожидание потребителя**  
   Перед началом передачи производитель проверяет, что `/test_full` инициализирован (пока потребителсь не запустится), и ждёт готовности.
3. **Передача данных**  
   - Производитель:
     - минимизирует `sem_empty_` и записывает данные в ячейку буфера;
     - инкрементирует `sem_full_`, сигнализируя потребителю.
   - Потребитель:
     - минимизирует `sem_full_` и читает данные из ячейки;
     - инкрементирует `sem_empty_`, освобождая место для следующего элемента.
4. **Завершение**  
   После завершения обмена ресурсы корректно закрываются (`sem_close`, `munmap`, `close`), но **удаление объектов (`shm_unlink`, `sem_unlink`) не выполняется**, чтобы не блокировать запуск потребителя.

## Выбор алгоритма сжатия

В данном проекте **сжатие** реализовано как **квантование `float` в `int16_t`** с масштабным коэффициентом (`scale_`). Такой подход:

- обеспечивает **предсказуемую потерю точности**, которая зависит от `scale_`;
- позволяет **сильно сократить объём данных** за счёт уменьшения разрядности;
- работает **быстро**, без сложных алгоритмов (например, `zlib`/`LZ`), что важно для **канала с низким латентсенсом**.

Ход мыслей при разработке:
- выбор `int16_t` обусловлен компромиссом между **размером** и **точностью**;
- `FloatQuantizer` обеспечивает **отсечение** значений за границами `int16_t` и **линейное масштабирование**;
- однократный пропуск `float` → `int16_t` → `float` даёт **предсказуемую погрешность**, которую можно измерять (`LossPercent`).

## Компромиссы и оптимизации

- **Размер элемента кольцевого буфера**  
  Фиксированный размер `kItemSize` (заголовок + `kMaxFloatPerChunk * 2`) упрощает синхронизацию и уменьшает накладные расходы на пересылку метаданных. Однако это ограничивает гибкость при работе с данными переменного размера.
- **Потеря данных**  
  Квантование `float` в `int16_t` с масштабом допускает **некоторую потерю точности**, но обеспечивает гораздо **меньший размер трафика** по сравнению с передачей `float` целиком.
- **Синхронизация**  
  Использование **именованных семафоров** и `shm_open` позволяет эффективно синхронизировать процессы без активного опроса (`busy‑wait`), но требует корректного управления жизненным циклом объектов (`shm_unlink`/`sem_unlink`).
- **Очистка ресурсов**  
  Для упрощения логики запуска/остановки объекты shared memory и семафоров не удаляются в `Cleanup()` в рабочем варианте; при необходимости их можно удалять отдельно в CI‑скриптах или вручную через `shm_unlink` и `sem_unlink`.
